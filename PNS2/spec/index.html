PNS2

StatusDraft / Reference

Section 1

Overview

PNS 2.0 is a Substrate-based on-chain naming system that acts as the authoritative zone source for a standards-compliant DNS service. A fronting DNS server (the Snorkel) translates chain state into UDP/53 responses using ordinary DNS wire format and ordinary record types. Recursive resolvers, stub resolvers, and existing client libraries interact with PNS names exactly as they would with any traditional DNS zone.

What's novel is the zone source: zones live in Substrate storage, edits are extrinsics, and every record is a signed state transition with on-chain economics — registration fees, grace periods, deposit-driven cleanup. What's ordinary is everything downstream: the Snorkel speaks DNS, not anything custom.

          ┌─────────────────┐   ┌─────────────────┐    ┌─────────────────┐
dig  ───▶│ recursive       │──▶│ snorkel         │◀──▶│ pns-node        │

          │ (1.1.1.1 etc.)  │   │  server)        │    │  zone source)   │
          └─────────────────┘   └─────────────────┘    └─────────────────┘
                        UDP/53            JSON-RPC
                        DNS wire          (pns_* namespace)
                        format            + state subscription

1.1 Historical Note

An earlier project also called "PNS" was authored by cupnfish and collaborators around 2021, incomplete and abandoned since. PNS 2.0 is a new ground-up implementation sharing only the product name and general intent. It does not derive code, types, or storage layouts from that earlier work. This document is the first complete written specification.

1.2 Scope

This specification covers: the on-chain data model (names, records, ownership, subdomains, listings); the six FRAME pallets that comprise the runtime; the RPC and runtime API surfaces; the DNS record-type registry; the economic model (registration pricing, fee split, cleanup incentives); and the Snorkel's role as a standards-compliant DNS front-end.

Out of scope: internal implementation of the Snorkel's hot path; gateway/reverse-proxy indirection for web content; integration with specific recursive resolvers; wallet UX for name registration flows.

Section 2

Design Goals

  • Chain is authoritative. Every name, every record, every ownership transition is a transparent on-chain fact. No off-chain state.
  • Standards compliance end-user side. Clients do not need to know PNS exists. dig alice.dot works. nslookup alice.dot works. No custom client protocol.
  • Economically self-sustaining cleanup. Expired names do not require a privileged purge process. A deposit-based cleanup incentive lets any actor reap expired names and recover the protocol's storage cost.
  • One canonical identity per account. Each on-chain account holds at most one top-level name; subdomains are the delegation mechanism.
  • Monolithic name binding. A name binds to its owner's account; ownership transfer rotates records. No in-flight key changes.
Section 3

Architecture

3.1 Three-Layer Model

LayerComponentRole
3Gateway (optional)HTTPS reverse proxy that resolves names via Snorkel and serves CONTENT records (IPFS CID to web content). Out of scope for this spec.
2Snorkel (DNS server)UDP/53 authoritative responses (RFC 1035 wire format). Zones populated from pns-node via pns_* JSON-RPC. Built-in janitor. Rate limiting and merkle-verified cache.
1pns-node (Substrate solochain)PnsRegistrar, PnsResolvers, PnsMarketplace, PnsNft, PnsPriceOracle, PnsRegistry pallets. Authoritative state; all writes are extrinsics.

3.2 Data Flow — Resolution

  1. 1. Client (dig, browser) queries the Snorkel on UDP/53 for alice.dot IN A.
  2. 2. Snorkel computes the namehash of alice.dot and issues pns_lookup(namehash, [1]) to pns-node.
  3. 3. pns-node reads Records<T> storage map for that namehash, returns the IPv4 bytes.
  4. 4. Snorkel formats a standard DNS response and writes it to the UDP socket.

3.3 Data Flow — Registration

  1. 1. User constructs a SCALE-encoded register(name = b"alice", reject_offer = None) extrinsic.
  2. 2. User signs with their SS58 account and submits via wallet or polkadot-js.
  3. 3. pns-node validates: label legal, name not reserved, not already owned, caller has no existing canonical name, sufficient balance.
  4. 4. Registration fee is computed and settled; a 5% cleanup deposit is held on the caller; the NFT mints; NameRegistered event fires.
  5. 5. Next block, the Snorkel observes the event, refreshes its cached zone; subsequent DNS queries for alice.dot resolve correctly.

3.4 Namespace Structure

Every name in PNS 2.0 resolves under a single base TLD (the "base node"):

  • dot. — canonical PNS 2.0 TLD when built without the kusama feature flag
  • ksm. — canonical PNS 2.0 TLD when built with the kusama feature

These constants (DOT_BASENODE, KSM_BASENODE) are defined as pre-computed sp_core::H256 values in pallets/pns-types/src/lib.rs. The NATIVE_BASENODE alias selects one at compile time.

For the DNS-visible surface: alice registered on-chain is queryable as alice.dot. sally.alice is queryable as sally.alice.dot.

For the substrate.icu deployment, names are visible under *.paseo.substrate.icu on Paseo testnet. The dot. TLD structure applies to the standalone pns-node; the Snorkel bridges these to the substrate.icu zone.
Section 4

Naming

4.1 Labels

A label is one level of the DNS hierarchy (e.g., alice in alice.dot).

RuleValue
Length1–63 bytes
Character setASCII alphanumeric (a–z, 0–9) after lowercasing
Case sensitivityNone — Alice, alice, and ALICE are the same name
Allowed punctuationNone (no hyphens, no underscores)

Labels that fail validation are rejected with LabelInvalid or ParseLabelFailed at extrinsic-call time.

4.2 Canonical Names and Subdomains

A canonical name (top-level registration) is <label>.<base>, e.g., alice.dot. Canonical names are minted by the register extrinsic on PnsRegistrar.

A subdomain is one level deeper: <sub>.<label>.<base>, e.g., sally.alice.dot. Subdomains are created by delegation from the parent's owner via offer_subdomain → accept_subdomain.

Only two levels of hierarchy are supported. foo.bar.alice.dot will fail parse_name_to_node.

4.3 Namehash

PNS 2.0 uses the ENS-style recursive Keccak256 namehash:

hash_label(L)     = Keccak256(lowercase(L))
encode(parent, h) = Keccak256(SCALE(parent) || SCALE(h))

namehash("alice")     = encode(BASENODE, hash_label("alice"))
namehash("sub.alice") = encode(namehash("alice"), hash_label("sub"))

All on-chain state keyed by a name uses this 32-byte DomainHash (sp_core::H256). Extrinsics and RPC methods accept plain Vec<u8> label bytes and compute the hash internally — callers do not construct namehashes directly.

4.4 Reserved Names

A reserved list blocks registration of certain labels at the protocol level. Genesis reservations include ecosystem-brand terms and first-party product names (approximately 48 entries). The list is managed by two manager-only extrinsics: add_reserved(name) and remove_reserved(name). Attempting to register a reserved name returns Error::Frozen.

Ownership accounting + subdomain delegation. One public extrinsic: set_official(official) (Manager only) — transfers the base-node NFT (the TLD itself) to a new account for operational handover.

6.6 PnsNft (index 8)

ERC-721-style NFT base. Token ID is the 32-byte namehash; class ID 0 is reserved for PNS names. Not directly called by end users — ownership transfers happen via PnsRegistrar extrinsics that invoke this pallet internally.

7.3 Text Metadata

Stored separately under Texts<T>. Known kinds: Email, Url, Avatar, Description, Notice, Keywords, Twitter, Github, Ipfs. Texts are not cleared on ownership transfer.

7.4 Record Lifecycle on Ownership Transfer

When a name changes owner, all non-SS58 and non-ORIGIN records are cleared. This prevents a new owner from inheriting stale technical state: endpoints pointing to the former owner's infrastructure, validator stashes on the old owner's nodes, or public keys tied to the old owner's cryptography. Texts are retained.

Section 10

Economic Model

10.1 Fee Distribution — Register / Renew

ShareRecipientForm
5%RegistrantHeld under HoldReason::CleanupDeposit. Refundable via cleanup bounty or release on renew.
40%Block authorTransferred to the block author of the containing block (or PnsCustodian if unavailable).
55%PnsCustodianTreasury / protocol account; governance-controlled SS58.
The 5% is not a fee — it's a hold on the registrant's balance that becomes the reward for whoever eventually cleans up the expired name. If the registrant renews before expiry, the old hold is released and a new one taken against the renewal fee.

10.2 Cleanup — Permissionless Janitor

The PnsRegistrar::cleanup(deadline) extrinsic is the protocol's self-sustaining expiry-reap mechanism. Any account can call it. The caller earns the cleanup deposits held on every expired name they reap.

  • deadline must equal expire + GracePeriod of the names being reaped, and must be <= now or the call errors with NotExpired
  • Iterates ExpiryIndex at deadline, taking up to 5 namehashes per call (bounded weight)
  • For each name: releases 5% hold on original registrant, transfers it to caller, removes OwnerToPrimaryName entry, force-deletes NFT and all subname records, removes from ExpiryIndex
  • Emits NamesCleaned { count, caller, payout }
  • Deadlines with more than 5 expired names require multiple calls — competitive reaping is expected

10.3 TTL and Grace Period

Every registration expires at mint_time + MaxRegistrationDuration (365 days). Between expiry and expire + GracePeriod (30 days past expiry): DNS lookups return null; only the previous owner can renew; cleanup is not yet callable.

After expire + GracePeriod: renew() fails; cleanup(deadline) becomes callable by anyone; the label returns to the free pool.

10.4 Marketplace Economics

  • Listing deposit: held on seller while listing is active; released on cancellation, sale, or permissionless cleanup_listing
  • Buyer payment on sale: seller_price + full_registration_fee. The registration fee portion goes through the same 5/40/55 split
  • Protocol fee: 2% of the seller's asking price; separate from the registration-fee split
Section 11

The Snorkel (DNS Server)

The Snorkel is a standards-compliant authoritative DNS server with a chain-aware backend. It speaks UDP/53, uses RFC 1035 wire format, and returns ordinary A/AAAA/CNAME/TXT records. What's different is the zone source: each query translates into pns_* JSON-RPC calls against a pns-node, rather than a zone file.

11.1 DNS Compatibility

FeaturePNS 2.0 / Snorkel
Wire formatRFC 1035 binary; no extensions required
TransportUDP/53 authoritative (TCP fallback for large answers)
Record typesStandard A, AAAA, CNAME, TXT plus custom types in the 65280+ private-use range
Response codesStandard NOERROR / NXDOMAIN / SERVFAIL / REFUSED
TTLDerived from grace-period math against a configured maximum
Authoritative bitAlways set; Snorkel is the authoritative server for its configured zones
A recursive resolver configured to forward dot. queries to a Snorkel instance does not need any PNS-specific code.

11.2 Resolution Hot Path

  1. 1. Parse the DNS request (name + type) from the UDP datagram
  2. 2. Compute the PNS namehash of the name
  3. 3. Issue pns_lookup(namehash, [requested_type]) to pns-node via JSON-RPC
  4. 4. Translate returned (RecordType, Bytes) tuples into DNS answer-section records
  5. 5. Write the response to the socket

The Snorkel caches results (merkle-verified against chain state heads). When NameRegistered / RecordsChanged events are observed on the chain, relevant entries are invalidated.

11.3 Hot-Path Properties

  • Per-worker geometry: each CPU core runs an independent request-handling pipeline; no shared mutable state on the hot path
  • Two-process IPC isolation: the chain-RPC process is separate from the DNS-serving process; a panic in one cannot take down the other
  • Panic-free by construction: the hot path lints against unwrap / expect; errors return SERVFAIL cleanly
  • Response rate limiting: a jittered RRL defends against traffic-amplification abuse
  • Merkle-verified cache: cached entries are authenticated against chain-state roots

11.4 Built-in Janitor

The Snorkel already subscribes to registrar events to keep its cached zones fresh, making it the natural operator of the permissionless cleanup path. Its built-in janitor works as follows:

  • The chain-state monitor observes names whose expire + GracePeriod has passed
  • The janitor constructs a cleanup(deadline) extrinsic
  • The extrinsic is submitted signed by a configurable "janitor beneficiary" SS58 address — whoever is configured earns the released deposits
  • The Snorkel emits a log record of the cleanup (txhash, reaped count, payout) for operator audit
cleanup() is permissionless at the pallet level. The Snorkel is one possible caller, not a privileged actor. Anyone can run their own cleanup process. The Snorkel's advantage is that it's already watching the relevant events.

11.5 Status

As of this spec version, the Snorkel is in active development. The pns-node side (chain, pallets, RPC, runtime API) is runnable on Paseo testnet. Readers should verify against the Snorkel project's own documentation for current implementation status.

Section 12

Invariants

Load-bearing rules that every pallet must uphold. Violations are bugs.

  • One canonical name per account. OwnerToPrimaryName<T> is 1:1. Enforced at register, transfer, and accept_offered_name.
  • Names are case-insensitive. Labels are lowercased before Keccak256 hashing; Alice and alice collide to the same namehash.
  • Names are monolithic. A name's cryptographic identity (namehash) is fixed at registration. No key rotation; no identity mutation.
  • Two-level hierarchy only. Only one level of subdomain is supported. foo.bar.alice.dot is rejected.
  • Record ownership propagates with NFT. Only the current NFT owner may set records.
  • SS58 and ORIGIN are protocol-managed. User-issued set_record on these types fails with Ss58RecordProtected. SS58 rotates on ownership transfer; ORIGIN is write-once at register.
  • Non-SS58 records clear on ownership transfer. Prevents stale technical state from following a name.
  • Fees and deposits are settled atomically. The 5/40/55 split is computed and settled in a single extrinsic execution.
  • Cleanup is weight-bounded. At most 5 names per cleanup() call.
  • Reserved names cannot be registered. ReservedList<T> consulted at register-time; a match errors with Frozen.
Section 13

Security Considerations

13.1 Replay and Offer Freshness

Gift offers expire after OfferWindow (90 days). Beyond that window the offer becomes invalid and the name returns to the free pool. Prevents indefinite-pending gift state.

13.2 Stale-Entry Cleanup

Auto-cleanup of OwnerToPrimaryName on register prevents accounts from being permanently blocked from re-registering after a prior name's expiry. Full state eviction requires a cleanup call.

13.3 Record-Clearing on Transfer

Non-SS58 records are cleared on ownership transfer. New owners do not inherit stale endpoints; relying parties querying the new owner see a clean slate.

13.4 Grace-Period Semantics

During grace, DNS lookups return null — the name appears unowned. This is deliberate: relying parties treat names in grace as already-gone, even though the protocol allows the previous owner to renew. Prevents a near-expired name from being treated as live past its lifetime if renewal fails.

13.5 Weight-Bounded Operations

Cleanup batches are bounded at 5 names per call. Permissionless callers cannot DoS the chain by staging large reap cohorts. Registration, transfer, and record writes are O(1) in state size.

13.6 Reserved List as Brand Protection

The genesis reserved list reserves ecosystem-brand terms (polkadot, kusama, assethub, fellowship, etc.) and first-party product names (snorkel, zkpki, dotwave, etc.) from open registration. Additions and removals go through the manager origin.

Section 15

Out of Scope / Future Work

  • Multi-level subdomains beyond depth 2. Current parse_name_to_node supports exactly one dot. Arbitrary depth would require recursive parsing and storage model changes.
  • Per-record TTLs. Currently a single Snorkel-computed TTL per name.
  • DNSSEC signing chains. The chain provides cryptographic proof of ownership, but Snorkel DNS responses are not currently DNSSEC-signed.
  • Reverse records (PTR). No current support for reverse IP to name lookup.
  • IDN / punycode. Labels are ASCII-only. Internationalized names would require punycode normalization before hashing.
  • PQ-secure namehash. Keccak256 is not post-quantum compromised, but a schema-versioned upgrade path would let a future PQ hash be adopted without breaking existing names.
  • Rent-pricing model. set_rent_price exists with a zero-valued tier table; future designs could bill ongoing storage rent.
  • Gateway layer. The reverse-proxy / content-serving layer (Layer 3) is its own project; PNS 2.0 provides the CONTENT record but does not mandate the gateway's implementation.
